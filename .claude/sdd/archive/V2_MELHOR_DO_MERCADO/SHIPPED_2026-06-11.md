## Lições Aprendidas

### ✅ O que funcionou bem

**SQLAlchemy JSON vs JSONB pattern**
Usar `JSON` nos modelos e `JSONB` só nas migrations permite rodar SQLite nos testes de unidade e PostgreSQL em produção sem duplicar modelos. É o padrão correto para este projeto.

**`dependency_overrides[get_current_user]` nos testes de integração**
Sobrescrever diretamente o dep `get_current_user` é mais limpo e robusto do que patchar `firebase_auth` + adicionar header `Authorization`. Evita o problema de `HTTPBearer` rejeitar requests sem header antes de chamar o mock.

**`consume()` não deve chamar `db.commit()`**
Funções de serviço não devem gerenciar o ciclo de vida da transação. Quem chama decide quando commitar — no FastAPI isso acontece automaticamente ao fim do request; nos testes, o rollback no teardown limpa os dados. Violar isso causa contaminação entre testes em engine module-scoped.

### ⚠️ O que melhorar

**UUID como PK em raw SQL**
O `default=new_uuid` do SQLAlchemy é Python-level e não gera `DEFAULT` na DDL. Qualquer INSERT raw que omita o `id` falha com NOT NULL em SQLite (PostgreSQL tolera porque `gen_random_uuid()` pode estar como default no servidor). Sempre incluir `id` explicitamente em raw SQL, ou usar `server_default=text("gen_random_uuid()")` na coluna.

**Fixtures function-scoped + engine module-scoped**
Quando um teste chama `db.commit()` (ex: para testar lógica de expiração), os dados das fixtures se tornam permanentes no engine module-scoped. O padrão mais seguro: ou todos os testes usam engine function-scoped, ou fixtures que precisam de commit usam IDs totalmente únicos e as assertions são agnósticas ao código exato.

**Importar `unused` no `test_subscription_lifecycle` após refactor**
Remover `from unittest.mock import patch` ao refatorar para `dependency_overrides` mas deixar uma `with patch(...)` sobrevivente causou `NameError` tarde. Grep por `patch(` deve ser parte do checklist de limpeza de imports.
